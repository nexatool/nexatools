# Contributing to NexaTools

Thank you for your interest in contributing to NexaTools! 🎉

NexaTools is a collection of free, browser-based tools built to help students, freelancers, and businesses without logins, paywalls, or clutter. Whether you're fixing a typo, squashing a bug, or proposing a new tool, your contributions are welcome and appreciated.

This project is maintained by **Rashid Amin**.

---

## 🐛 Reporting Bugs

If you find a bug, please help us fix it by opening an issue with as much detail as possible:

- A clear, descriptive title
- Steps to reproduce the issue
- What you expected to happen vs. what actually happened
- Screenshots, if applicable
- Your browser and device (e.g. Chrome on Windows, Safari on iOS)
- Which tool the bug affects (e.g. Resume Builder, Invoice Generator)

Before opening a new issue, please search existing issues to avoid duplicates.

**Note:** For security vulnerabilities, please do **not** open a public issue. Refer to our [SECURITY.md](./SECURITY.md) for responsible disclosure instructions.

---

## 💡 Suggesting Features

Have an idea for a new tool or an improvement to an existing one? We'd love to hear it.

When suggesting a feature, please include:

- A clear description of the feature or tool
- The problem it solves or the use case it supports
- Any examples, mockups, or references that help explain the idea
- Whether you'd be interested in helping build it

Feature suggestions can be submitted as a GitHub issue, labeled as a feature request where possible.

---

## 🔧 Pull Request Guidelines

1. **Fork** the repository and create your branch from `main`:
   ```
   git checkout -b feature/your-feature-name
   ```
2. **Keep changes focused** one feature or fix per pull request makes review easier and faster.
3. **Test your changes** locally before submitting to make sure nothing is broken.
4. **Write clear commit messages** that describe what changed and why.
5. **Update documentation** (README, comments, etc.) if your change affects how something works or is used.
6. **Open a pull request** with:
   - A descriptive title
   - A summary of what the PR does and why
   - Any related issue numbers (e.g. `Fixes #12`)
7. Be responsive to feedback during review small revisions are a normal part of the process.

Pull requests that are unrelated to an existing issue or discussion are still welcome, but please describe your reasoning clearly so it can be reviewed properly.

---

## 🎨 Code Style Expectations

To keep the codebase consistent and easy to maintain:

- Write clean, readable HTML, CSS, and JavaScript.
- Use consistent indentation (spaces, not tabs, unless the existing file uses tabs).
- Keep functions and components focused and reasonably sized.
- Avoid unnecessary dependencies  NexaTools favors lightweight, simple solutions.
- Comment non-obvious logic so others can follow your reasoning.
- Match the existing naming conventions and file structure used in the project.
- Remove debug code, console logs, and commented-out blocks before submitting a PR.

If you're unsure about a style choice, look at existing code in the repository as a reference, or ask in your PR description.

---

## 🤝 Respectful Collaboration

NexaTools is built for everyone, and we want contributing to it to feel welcoming for everyone too.

- Be respectful and constructive in issues, discussions, and code reviews.
- Assume good intent disagreements happen, but they should stay focused on the work, not the person.
- Be patient with maintainers and other contributors; this project is maintained on a best-effort basis.
- Harassment, discrimination, or disrespectful behavior of any kind will not be tolerated.

If you have concerns about behavior in the project, please reach out via [nexatools.io/contact.html](https://www.nexatools.io/contact.html).

---

Thank you for helping make NexaTools better for everyone. We're glad you're here! 🙌
